Threadwise analytics: the founder's operating sheet

Use this for the ten-minute weekly review: what each signal means, what it does not mean, and what decision it should trigger.

Read the lesson Open the live dashboard

The mental model

Threadwise emits receipts → PostHog groups the receipts → you form a diagnosis → you change one thing → the next receipts tell you whether it helped.

Weekly review order

1

Reliability

Read successful writes, failed writes, and retry outcomes first. If the product cannot reliably finish the job, do not optimize polish yet.

2

Use

Check queue opens, reviews started, and batches completed. These show whether the workflow is actually being used repeatedly.

3

Quality

Check approve/edit/reject mix. Today this reflects rule-proposal decisions; add explicit classification decisions before treating it as label accuracy.

4

Speed

Check decision and completion duration only after enough real review sessions exist to make the distribution meaningful.

Signal → diagnosis → action

SignalLikely diagnosisNext action
Queue opens are low while backlog existsThe queue is hard to find or not worth entering.Improve the Home prompt or queue entry point.
Reviews start but decisions are lowThe proposal is confusing, slow, or demands too much thought.Inspect copy and reduce decision complexity.
Edit/reject share risesRule interpretation may be weak.Review examples and improve rule generation or taxonomy.
Write failures riseThe provider execution path is unhealthy.Fix reliability before adding features.
Retries usually recoverThe failure is transient and retry UX is valuable.Keep retry visible; investigate the transient cause.
Batches rarely completeThe queue may be too tedious or too large.Improve batching, shortcuts, or matching-rule leverage.

What to add next

Anonymous review session

Generate a fresh safe ID when a review queue begins and carry it through the write. This makes repeated one-user funnels meaningful.

Classification decision

Separate approve/edit/reject of the proposed label from approval/edit/reject of the broader rule.

Label and source properties

Add proposed label, final label, suggestion source, and confidence bucket. These aggregate cleanly and reveal where quality is weak.

Do not add

Do not dump email bodies, subjects, or arbitrary rule prose into PostHog. Even if privacy is not a concern, each value is unique, so it produces expensive noise rather than useful breakdowns.

Mnemonic

R-U-Q-S

Reliability first. Use second. Quality third. Speed fourth.
