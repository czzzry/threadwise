One-Sentence Definition

Async in Threadwise means the founder sees a useful result as soon as possible, while slower supporting work continues separately and reports back through visible state changes instead of one long opaque wait.

Mnemonic

A-W-D-R

Acknowledge the action quickly.

Work in visible stages.

Deliver the main result first.

Refresh slower follow-up state in the background.

State Machines

Selected email

reading -> understanding -> ready

Purpose: reduce the frozen-feeling gap when Gmail selection changes.

Teach / fix

previewing -> applying -> result or retry

Purpose: separate request acceptance from final completion.

Follow-up refresh

working -> done or retry

Purpose: move summary warming and broader refresh off the founder's main path.

Architecture Map

Gmail page
  -> extension content script
      -> local companion API
          -> local storage + teaching loop + Gmail write-through
              -> returns fast sidebar state
                  -> background follow-up refresh
                      -> extension poll picks up done/retry state

What Changed In Code

Area Main code Why it matters
Selected-email understanding src/gmail_companion_state.py Introduces the `reading / understanding / ready` contract for current-email comprehension.
Server-side async follow-up src/gmail_companion_ui.py Returns the fast current-email result, then refreshes heavier summary state in a background thread.
Reusable rule data src/teaching_loop.py Reuses already-computed semantic rule data during apply instead of rebuilding preview work.
Founder-visible state machine extensions/gmail_companion/content.js Shows accepted, working, done, retry, and compact recent activity in the extension.
Proof tests/test_gmail_companion_ui.py Protects the architecture through product-facing tests instead of relying on memory.

Why Extension First Was The Right Call

What the founder felt

Threadwise seemed slow or frozen while understanding an email and doing work after a teach/fix action.

What we diagnosed

The pain came from the interaction model and state visibility, not from the extension shell itself.

Decision

Keep the extension as the primary entry point and redesign the request lifecycle first.

When Professionals Use Async

Use async when... Do not reach for async when...
The user would otherwise wait on network I/O, browser messaging, storage reads, summary recomputation, or background refresh work. The work is trivial, deterministic, and finishes instantly enough that extra state machinery would only add complexity.
You can deliver a smaller meaningful result now and a richer result later. The action must remain strictly atomic and showing an intermediate result would be misleading.
You need the UI to distinguish accepted, working, done, and retry. You have no intention of exposing state changes and the added queueing would become hidden complexity.

Truthful Resume / Demo Talking Points

Strong claim

I redesigned a human-in-the-loop browser extension flow from opaque blocking interactions into staged async state machines with background follow-up refresh and retry visibility.

Why it is true

The implementation now has explicit lifecycle states in both the local companion contract and the extension UI, plus tests proving the contract.

Do not claim

Do not say we built a distributed queueing system or a standalone app. We did not.
