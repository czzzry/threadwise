A before/after architecture map for the completed bounded refactor: reusable batch workflow behavior now lives outside CLI files, Gmail-specific review behavior is explicit, and product behavior was preserved.

Implementation completed for issue 054 Current docs read: AGENTS, CONTEXT, V2 alignment, PRD, checkpoint Validation: 296 unittest tests passed
What Is Healthy 1

The repo already has a clear product spine: fetch mail, normalize it, classify it, store local evidence, report what happened, and only mutate Gmail inside bounded rules.

Main Problem 2

Some reusable business behavior lives inside command-line modules, so other commands import private helpers from CLI files instead of stable application modules.

Recommended Scope 3

A small architecture refactor focused on workflow organization, not classifier behavior, Gmail OAuth, or a broad repo cleanup. This has now been implemented.

Implementation Plan

This is the issue-sized plan that was implemented. Each step was behavior-preserving and validated with the available local tests.

Characterize Current Behavior

Verified tests around current command outputs and artifact contents for Gmail daily run, ProtonMail daily run, write status, inbox removal status, and daily reports.

Extract Shared Path And Report Helpers

Move repeated path resolution and daily report writing into small shared modules. Commands should call these helpers instead of each owning near-duplicate code.

Extract Gmail Review Support From CLI

Move reusable Gmail review helpers out of live_gmail_review_cli.py into a module with a library name. The CLI keeps only parsing, prompting, and command-specific output.

Make Gmail-Specific Store Behavior Explicit

Separate generic stored-batch persistence from Gmail raw-message refresh and reclassification. This makes the provider-aware model easier to trust.

Run Tests And Compare Artifacts

Ran full local unittest discovery. The refactor is considered stable because all 296 discovered tests passed.

What Stays Out

These are real areas, but they are not the first refactor.

Classifier rewrite: fixture_classifier.py is the biggest file, but changing it risks label behavior. Save it for a later classifier-specific plan.
Gmail OAuth split: live_gmail_client.py mixes OAuth and Gmail calls, but touching it first increases risk around credentials and live access.
Browser UI redesign: the local workbench is large but not the current bottleneck for daily-use readiness.
New product behavior: no new Gmail actions, no ProtonMail writes, no scheduling, no broader automation.

Decision Summary

recommended

Completed one bounded refactor issue focused on command/workflow organization and provider-aware review storage.

why now

The repo was waiting on new email evidence, so this reduced structural confusion without changing product scope or touching live inbox behavior.

approval gate

The implementation brief is recorded in docs/issues/054-refactor-batch-workflow-command-boundaries.md. No live inbox commands were run.
