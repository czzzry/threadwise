# Hermes Project Brief

## What this product is

Threadwise is a local-first email-agent product focused on supervised inbox triage before provider-side action.

Today the repo primarily delivers:

- a Gmail companion sidebar and local companion server
- selected-email explanation and correction flows
- bounded Gmail label write-back and limited low-value `INBOX` removal
- local review, reporting, teaching, and audit artifacts
- ProtonMail read-only fetch/reporting paths

This repo is not a generic autonomous email operator and should not be reframed that way.

## Primary user flows

1. Open Gmail with the companion enabled and inspect the selected email.
2. Review the agent's classification, rationale, and suggested next step.
3. Correct or teach the agent in context.
4. Preview broader impact before applying a change to similar emails.
5. Approve bounded provider-side actions only when the flow explicitly calls for them.
6. Use the dashboard, review workbench, daily reports, and unsubscribe review surfaces for follow-up and auditing.

## UI and UX direction

- Preserve the current Threadwise identity: warm, editorial, human-guided, and intentionally non-corporate.
- Keep the product feeling local, supervised, and trustworthy rather than fully autonomous.
- Favor clarity over density.
- Keep correction, review, and confirmation states explicit.
- Avoid introducing generic enterprise UI, generic chatbot framing, or visual churn unless a task explicitly requests a redesign.

## Safety constraints

- Do not expand the product into broad delete, trash, send, reply, or autonomous archive actions by default.
- Do not silently increase provider-side mutation scope.
- Treat credentials, OAuth, private email content, unsubscribe execution, and live inbox actions as sensitive.
- ProtonMail should remain read-only unless the founder explicitly changes that boundary.
- Historical docs exist in this repo; verify the current source of truth before steering implementation.

## How Hermes should work with Codex

- Hermes is the operator and coordinator.
- Codex is the implementation worker.
- Hermes should delegate bounded coding, testing, refactoring, and repo-inspection tasks to Codex.
- Hermes should pass Codex a narrow implementation brief, expected constraints, and what to verify.
- Hermes should ask Codex for small diffs and avoid bundling multiple unrelated changes into one request.
- Hermes should keep product-direction decisions, sequencing, and approval collection on the Hermes side unless explicitly delegated.

## What Hermes must verify after Codex changes code

- The change matches the requested slice and did not widen scope.
- Current UI tone and aesthetic were preserved unless change approval said otherwise.
- No new dependency was added without approval.
- No destructive email/provider action path was introduced or widened.
- The relevant checks were run for the touched area.
- The summary includes changed files, validation, remaining risks, and a sensible next step.
