Lesson 0002 · Architecture + observability

PostHog watches Threadwise. It does not run Threadwise.

Your tangible win: after this lesson, you should be able to trace one user action from Gmail to PostHog, name exactly what crosses the boundary, and explain why analytics failure cannot block an email review or Gmail label write.

Previous: async architecture Open the PostHog reference sheet

The answer first

PostHog adds an observability side channel; it does not change Threadwise's decision architecture. The extension and local companion already had to communicate. We attached small, privacy-checked event messages to that existing boundary. The core path still works without PostHog.

Start with the three planes

1. Product plane

Gmail selection, Threadwise classification, review UI, teach/fix decisions, stored rules, and Gmail label writes. This is the product doing work.

2. Observability plane

Small events such as “suggestion decision made,” plus safe properties such as decision_type=edit. This is evidence about the work.

3. Analysis plane

PostHog stores events so we can build trends and funnels. This is where we ask whether real users complete the intended path.

How we connected it, step by step

1

The extension declares meaningful moments

analytics.js exposes a tiny API: extension opened, queue opened, review started, suggestion decided, rule confirmed, and batch completed. content.js calls those functions at specific UI transitions. We chose manual custom events rather than generic click autocapture.

2

The extension strips detail before sending

The browser code constructs only approved properties. Counts become ranges such as 2-5; it sends the choice approve/edit/reject, not the email or the user's instruction. Unknown properties and strings resembling email addresses or authorization values are rejected in the browser.

3

The background worker crosses the local boundary

The Gmail content script cannot—and should not—hold the PostHog project token. It asks the extension background worker to POST to Threadwise's existing local companion at /api/analytics/capture.

4

The local companion validates everything again

ProductAnalytics is the only module allowed to call the PostHog SDK. It checks the event name, exact required and optional keys, enum values, numeric limits, anonymous ID format, and a prohibited-data list. This is the real trust boundary; browser validation is helpful but not sufficient.

5

The Python SDK sends to EU PostHog

The local .env supplies the project token, EU ingestion host, environment, and enabled switch. The companion creates the PostHog client with exception autocapture off, GeoIP disabled, and person profiles disabled per event.

What it deliberately does not read

No email content

No subject, sender, recipient, body, preview text, message ID, or thread ID.

No teaching prose

No rule text, LLM prompt, model output, or generated explanation.

No credentials

No OAuth value, authorization header, token, or Gmail identity.

No passive surveillance

No session replay, DOM autocapture, page-view capture, clickstream scraping, or exception stack capture in this implementation.

Architectural nuance: the extension itself can read the selected Gmail context because that is required for Threadwise to work. PostHog does not automatically inherit that access. Only the small event payload explicitly constructed by Threadwise crosses the analytics boundary.

Does this change our architecture?

Yes, structurally

We added an external service dependency, a local analytics adapter, a schema contract, anonymous installation identity, configuration, and an outbound network path.

No, operationally

Classification, sidebar state, teaching, Gmail writes, and retry behavior do not depend on PostHog responses. Analytics capture catches failures and returns false.

The best description

PostHog is a non-authoritative observer: architecturally present, operationally optional, never the source of truth.

Why this belongs with the async lesson

Async UI tells the founder what Threadwise believes is happening. PostHog tells the product team what users actually experienced across many interactions. Those are complementary feedback loops:

What PostHog can tell us—and what it cannot

Can answer

Of installations that opened a review queue, how many started a review, made a decision, confirmed a future rule, and completed the batch?

Cannot answer

Was a specific Amazon email correctly classified as Orders? That belongs to the eval set and live QA, because PostHog never receives the email or suggested category.

Cannot prove causality

If completions rise after a redesign, that is evidence—not automatic proof that the redesign alone caused the change.

Retrieval practice

Answer before opening the hints

  1. Why does the project token live in the local companion rather than the Gmail content script?
  2. If PostHog is unavailable, which Threadwise behaviors should stop?
  3. What is the difference between the extension reading an email and PostHog reading an email?
  4. Why do we bucket counts instead of sending exact counts?
  5. Which system judges classification quality: PostHog, tests, or the eval set?
Check your mental model

The token stays behind the local boundary; no core behavior should stop; access by one component is not automatic data transmission to another; buckets reduce unnecessary specificity; the eval set judges classification quality.

Your memory aid: S-I-D-E

S — Select a few meaningful product events.

I — Isolate the external SDK behind the local companion.

D — Deny every unapproved or sensitive property.

E — Ensure analytics failure never becomes product failure.
