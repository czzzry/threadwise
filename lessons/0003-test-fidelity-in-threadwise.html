Lesson 0003 · QA + architectural reasoning

A test proves only the path it actually exercised.

Your tangible win: you will be able to hear “all tests passed” and ask the one question that prevents false confidence: “Which real path did those tests exercise?”

Previous: PostHog as observer Open the test-fidelity reference

The answer

Test fidelity means how closely the test path matches the claim you want to make. A lower-fidelity test can be excellent evidence for logic and still be irrelevant to whether the real first screen opens.

The Threadwise evidence ladder

EvidenceWhat it actually exercisesWhat it can supportWhat it cannot support alone
Backend or unit testFunctions, state, and local service behavior“This logic behaves as specified.”“The Gmail extension opens and feels usable.”
Simulator browser testReal clicks against a safe, controlled Gmail-like page and stored data“This visible flow works in the simulator.”“Google's real page and the installed extension integrate correctly.”
Live Gmail harnessA real signed-in Gmail page with automated Threadwise injection and actions“This flow can work against live Gmail conditions.”“The founder's normal installed-extension entry path is identical.”
Founder click-throughThe normal installed product, real entry point, and human-visible experience“The actual founder path worked in this run.”Broad logic coverage or every edge case.

Higher is not universally “better.” Each layer catches different failures. The mistake is using evidence from one layer to make a claim about another.

Why your failure was possible

1

The shortcut exercised real code

Threadwise's simulator launches a safe local inbox and deliberately disables Gmail checks and Gmail write-through. It is useful and fast, but Google is absent from that path.

2

The claim quietly became too large

“The simulator flow passed” became “Threadwise works from a user's perspective.” The test result may have been accurate; the conclusion was not.

3

Your first click crossed an untested boundary

Your normal browser, installed extension, Gmail page, local companion, and current state all had to connect. A backend simulation could not detect a failure at that entry point.

A worked example

Claim

“A founder can click Edit suggestion in Gmail and see the correction flow.”

Insufficient evidence

A backend test calls the correction function and gets the expected result. That proves the correction logic, not the Gmail click or extension connection.

Matching evidence

Start from the normal Gmail entry point, click the actual control, observe the next state, and retain a screenshot or browser trace.

The important Threadwise nuance

The repo's live Gmail harness is more realistic than the simulator: it attaches to a real signed-in Gmail page. But it currently uses host-driven sidebar injection and message fulfillment. The project documentation explicitly says this differs from the founder's normal installed-extension path. So “LIVE” is valuable evidence, but the exact entry path must still be named.

Your four-question QA contract

  1. Environment: Was this backend-only, simulator, live Gmail harness, or my normal installed browser?
  2. Entry point: Where did you begin the test?
  3. Actions: What did you actually click or type?
  4. Evidence: What proves each step passed: assertion, screenshot, trace, or visible result?
