Lesson 0001

From "Is it frozen?" to a professional async interaction model

This lesson teaches the actual architectural move we made in Threadwise: we did not "make it async" in the abstract. We redesigned the founder experience so the product acknowledges work quickly, shows visible stages, finishes the main path first, and lets slower supporting work complete separately.

Mission Tie-In

You wanted to understand this well enough to speak truthfully on a resume and reason about future changes like a professional collaborator. That means we care about architecture, tradeoffs, and system behavior more than memorizing function names.

The User Problem We Actually Solved

What the founder experienced

When reviewing an email, Threadwise sometimes took long enough that it felt frozen or unclear whether it had accepted the action.

What would have been the wrong diagnosis

"Browser extensions are the problem. We need a standalone app now."

What we concluded instead

The bigger issue was the interaction model: too much work was hidden behind one blocking moment, and the UI did not reveal meaningful progress.

What "Async" Means In This Product

Not: "Use a trendy technical pattern because professionals do."

Not: "Rebuild the whole product as jobs, workers, queues, and dashboards."

Yes: "Split one long opaque interaction into visible stages so the founder gets confidence early and detail later."

Async here means we separate main-path value from secondary follow-up work.

The Four Loops We Now Have

1

Selected-email understanding loop

The extension no longer jumps from nothing to complete judgment. It moves through reading, then understanding, then ready.

This is implemented in src/gmail_companion_state.py and rendered in extensions/gmail_companion/content.js.

2

Teach / fix action loop

Submitting a correction now gives an immediate accepted state, then a visible working state, then either a done result or a retry surface.

This is the difference between "I hope it heard me" and "I know it accepted the request and I know what state it is in now."

3

Background follow-up loop

After the main result is returned, slower summary warming and broader refresh work continues behind the scenes and reports back as working, done, or retry.

This lives in the local companion in src/gmail_companion_ui.py.

4

Recent-activity loop

The founder can still see what Threadwise just did even after the original moment has passed. That keeps the sidebar compact without making it feel forgetful.

State Machine Diagram

Selected email
  idle
    -> reading
    -> understanding
    -> ready

Teach / fix
  draft
    -> previewing
    -> rule-proposed
    -> scope-confirmation
    -> applying
    -> result OR retry

Background follow-up
  working
    -> done OR retry

Why We Kept The Extension

Reason 1: right problem first

The user pain was waiting without confidence. A standalone app would not automatically solve that if the state model stayed opaque.

Reason 2: lower disruption

The extension is already the founder's working surface. We can improve the experience without forcing a new operating habit immediately.

Reason 3: reusable semantics

The async states we created are portable. A future standalone app can reuse them instead of starting from zero.

The Architectural Decision In Plain English

We chose to change the state model before changing the product shell.

That is a professional move because it separates the core behavior from the presentation vehicle. If the underlying interaction semantics are good, they can survive platform changes. If they are bad, a platform rewrite only hides the same confusion behind more effort.

When Professional Developers Reach For Async

Good async fit Poor async fit
The user is waiting on browser messaging, local I/O, summary recomputation, Gmail writes, or background refresh work. The work is instant, local, and simple enough that a richer state machine would only create complexity.
You can return partial value honestly before all follow-up work is done. The operation must be represented as one atomic decision and partial visibility would be misleading.
The user needs confidence that the request was accepted and to know what to do if it stalls. You have no plan to expose accepted / working / done / retry and would just bury the complexity.

Alternatives We Did Not Choose

Alternative: full standalone app now

Rejected because it attacked the shell before proving the better state model. Higher effort, weaker diagnosis.

Alternative: keep everything synchronous but add generic loading copy

Rejected because it still hides the lifecycle. The founder needs state semantics, not only a spinner.

Alternative: build a heavy queueing system immediately

Rejected because the MVP need was visible staged trust, not a full platform for background jobs.

Important Code-Level Decisions

A

We made understanding explicit

selected_email_understanding_state(...) adds a small timed state contract so the current email has a believable ramp from "I just clicked this" to "Threadwise is ready".

B

We made teach/apply request acceptance explicit

The content script now has local pending cards, disabled duplicate actions, and clearer result states.

C

We returned the fast path first

teach_apply() now starts a background follow-up refresh and returns a fast sidebar state instead of making the founder wait for secondary refresh warming.

D

We reused work instead of recomputing it

apply_sidebar_teaching() now returns semantic rule data so teach_apply() does not rebuild preview work just to continue the flow.

How To Explain This Like A Professional

"We kept the Gmail extension surface, but redesigned the interaction as explicit async state machines. The founder now gets immediate acknowledgment, visible progress, a fast main-path result, and a compact recent-activity/retry model while slower summary refresh work finishes in the background."
