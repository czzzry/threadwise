# Mission: Async Threadwise Extension

## Why
I want to understand the async Threadwise extension redesign well enough to speak truthfully about it on my resume, explain the architecture clearly, and collaborate on future product and engineering decisions like a professional developer instead of repeating memorized details.

## Success looks like
- I can explain the user problem, the architecture, and the state-machine changes in plain English without hand-waving.
- I can defend why we kept the Gmail extension and changed the interaction model first.
- I can reason about when async is the right tool here, when it is not, and what a future standalone Threadwise app could reuse from this design.

## Constraints
- The teaching must stay grounded in the actual Threadwise code and product decisions in this repo.
- The current product remains extension-first with super minimal UI in the product surface.
- I want zoomed-out explanations, diagrams, and memory aids, not isolated API trivia.

## Out of scope
- Building the full standalone Threadwise app right now
- Turning this into a generic JavaScript async course
- Polishing a large public teaching site before the product work is understood
